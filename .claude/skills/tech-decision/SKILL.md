### 规范2：技术选型评估维度

| 维度 | 权重 | 考虑因素 |
|------|------|---------|
| 团队熟悉度 | 高 | 学习成本、上手难度 |
| 社区活跃度 | 高 | 文档质量、问题解答 |
| 性能 | 中 | 是否满足性能需求 |
| 生态 | 中 | 插件、工具链 |
| 维护性 | 中 | 长期维护成本 |
| 扩展性 | 中 | 是否便于扩展 |

### 规范3：常见技术对比

#### Python Web 框架
| 框架 | 特点 | 适用场景 |
|------|------|---------|
| FastAPI | 异步、类型提示、自动文档 | API 服务、微服务 |
| Django | 全功能、ORM、Admin | 全栈应用、CMS |
| Flask | 轻量、灵活 | 小型项目、原型 |

**本项目选择 FastAPI 的理由：**
- 原生异步支持，适合 IO 密集型
- 自动生成 OpenAPI 文档
- 类型提示，代码更清晰
- 性能优秀

#### 前端框架
| 框架 | 特点 | 适用场景 |
|------|------|---------|
| Vue 3 | 渐进式、易上手、生态丰富 | 中后台、SPA |
| React | 函数式、灵活、生态最大 | 大型应用、跨平台 |
| Angular | 全功能、TypeScript | 企业级应用 |

**本项目选择 Vue 3 的理由：**
- 学习曲线平缓
- 中文社区活跃
- Element Plus 组件库成熟

#### 数据库
| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| MySQL | 成熟稳定、社区大 | 通用业务系统 |
| PostgreSQL | 功能强大、扩展性好 | 复杂查询、GIS |
| MongoDB | 文档型、灵活 | 非结构化数据 |

### 规范5：新技术引入原则

1. **解决实际问题** - 不为了新而新
2. **团队可接受** - 学习成本可控
3. **可回退** - 有备选方案
4. **小范围试点** - 先在非核心模块验证
5. **文档先行** - 记录选型理由和使用规范

## 常见问题

**Q: 要不要升级到最新版本？**
A: 评估：1) 新版本有什么改进 2) 是否有破坏性变更 3) 依赖是否兼容

**Q: 要不要引入新框架/库？**
A: 评估：1) 解决什么问题 2) 现有方案是否可行 3) 团队学习成本

**Q: 技术选型有分歧怎么办？**
A: 1) 明确评估标准 2) 小范围 POC 验证 3) 数据说话
