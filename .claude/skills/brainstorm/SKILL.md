# 头脑风暴技能

## 触发条件
- 关键词：头脑风暴、想法、方案、怎么做、建议、思路
- 场景：当用户需要讨论方案或寻求建议时

## 核心规范

### 规范1：需求分析框架

```
1. 背景：为什么要做这个功能？
2. 目标：要达成什么效果？
3. 用户：谁会使用这个功能？
4. 场景：在什么情况下使用？
5. 约束：有什么限制条件？
```

### 规范2：方案评估维度

| 维度 | 考虑因素 |
|------|---------|
| 可行性 | 技术是否可实现、时间是否充足 |
| 复杂度 | 实现难度、维护成本 |
| 扩展性 | 是否便于后续扩展 |
| 性能 | 是否满足性能要求 |
| 安全性 | 是否有安全风险 |
| 用户体验 | 是否易用、是否符合习惯 |

### 规范4：问题分析方法

#### 5W1H 分析法
```
What - 是什么问题？
Why - 为什么会发生？
Who - 谁受影响？
When - 什么时候发生？
Where - 在哪里发生？
How - 如何解决？
```

#### 鱼骨图分析
```
问题 ← 人员因素
     ← 流程因素
     ← 技术因素
     ← 环境因素
```

### 规范6：常见问题解决思路

#### 性能问题
```
1. 定位瓶颈（数据库？网络？计算？）
2. 分析原因（慢查询？N+1？大数据量？）
3. 制定方案（加索引？缓存？分页？）
4. 验证效果（压测对比）
```

#### 架构问题
```
1. 明确痛点（耦合？扩展难？维护难？）
2. 分析现状（现有架构图）
3. 设计目标（解耦？微服务？）
4. 渐进重构（分阶段实施）
```

#### 功能设计
```
1. 用户故事（作为xxx，我想xxx，以便xxx）
2. 功能列表（MVP 核心功能）
3. 原型设计（交互流程）
4. 技术方案（实现细节）
```

## 讨论原则

1. **先理解再建议** - 确保理解了问题再给建议
2. **多方案对比** - 提供多个方案供选择
3. **权衡利弊** - 说明每个方案的优缺点
4. **考虑约束** - 考虑时间、资源等限制
5. **渐进式** - 复杂问题分阶段解决

## 常用问题

**Q: 这个功能怎么实现？**
A: 先分析需求，再设计方案，最后评估工作量。

**Q: 用什么技术好？**
A: 根据场景、团队熟悉度、社区活跃度等因素综合考虑。

**Q: 这个问题怎么解决？**
A: 先定位问题根因，再针对性地制定解决方案。
